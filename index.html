Debt disclosures matter. The data is high value, legally messy, and rarely API-ready.

DebtSignal is a source-linked event model for financing and covenant disclosures buried in SEC 8-K filings and exhibits. The point is not to pretend legal prose is clean. The point is to return event records, provenance, and time metadata that help analysts and agents triage faster without masking uncertainty.

Why this dataset is valuable and messy

Credit-relevant facts often hide across multiple filing layers.

New facilities, amendments, waivers, defaults, guarantor changes, collateral changes, and maturity pushes can change credit risk quickly. But the facts rarely arrive as a single clean record. They may start in an 8-K item, continue in an exhibit index, and only become clear inside a credit agreement amendment, waiver letter, or indenture exhibit with issuer-specific language and redactions.

High signal

These disclosures affect diligence, surveillance, pricing, escalation, and issuer timeline work. They are operationally important even when only partially extractable.

High variance

The same concept can appear under different SEC items, agreement definitions, amendment numbering, exhibit labels, and issuer drafting habits.

High verification cost

Analysts still need the filing and exhibit trail. Agents need stable fields, explicit nulls, and provenance instead of polished but unverifiable summaries.

DebtSignal treats uncertainty as data: preserve source links, state time semantics, and leave ambiguous fields null.

What DebtSignal is

A docs-first contract for source-linked debt disclosure events.

The current product posture is a compact, machine-readable schema for financing and covenant-related events from public SEC filings. It is designed for monitoring, triage, prototyping, and evaluation. It is not presented here as a mature live extraction system or a substitute for source review.

What the model emphasizes

  • Issuer identity, filing metadata, event classification, and normalized terms where extractable.
  • Provenance objects and source excerpts so humans and agents can verify the claim path.
  • Timeline fields that separate filing time from event effective dates and text-derived date semantics.
  • Consistent envelopes and explicit machine-readable errors for integration-oriented workflows.

What this repo does not claim

  • No live production extraction backend is represented in this static site repo.
  • No claim of exhaustive issuer coverage, perfect term extraction, or continuous monitoring in production.
  • No legal parser, covenant compliance engine, or legal opinion workflow.
  • x402 is future-facing compatibility language only, not active billing or access control in this launch.

Agent-friendly schema principles

Stable envelopes, provenance-first records, explicit time and error behavior.

The contract is intentionally small. Responses use predictable data, meta, and page envelopes so agents can parse once and reuse logic across list and detail endpoints.

Stable envelopes

List responses return data plus meta and page. Detail responses return data plus meta. Envelope names do not change by endpoint.

Cursor pagination

page.next_cursor is opaque. Clients advance with the returned cursor rather than constructing offsets or inferring sort state.

Provenance first

Each event carries filing references, source excerpts, exhibit metadata, and extraction notes so downstream systems can verify before acting.

Time semantics

filed_at is the EDGAR filing timestamp. Event dates may be separate, nullable, and paired with precision fields when only calendar dates or narrative dates are known.

Explicit nulls

Absent or ambiguous terms stay null. Missing certainty is expressed through nullability, confidence, and extraction notes instead of synthetic defaults.

Machine-readable errors

400, 402, 429, and 503 use problem-detail objects with stable code values and retry hints where relevant.

Minimal endpoint set

Small, honest surface area for evaluation and integration planning.

The conceptual API stays narrow: list events, fetch one event, inspect event types, and discover the contract itself. The goal is integration clarity, not surface-area theater.

Roadmap note: 402 Payment Required is reserved in the contract for future paid access patterns, but this static launch does not implement x402 or live billing.
